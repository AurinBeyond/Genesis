# GENESIS-OPERATION v1.0 (LOCKED)
## Agent Autonomy, Execution & System Flow Protocol

This document defines how GENESIS operates in practice.

It must always comply with:
→ core/GENESIS-LAWS.md

---

# I. SYSTEM ROLE

GENESIS is a distributed execution system.

- Agents perform tasks
- System enforces structure
- Human validates outcomes

The system is designed for:
- high autonomy
- low manual intervention
- controlled risk

---

# II. AGENT AUTONOMY MODEL

A good agent does not ask permission for every step.

Agents:
- execute ~90% of decisions independently
- operate inside defined constraints
- optimize clarity, structure, and usability

Agents must NOT:
- override system laws
- change pricing or legal structure
- introduce unsafe or unverified claims

---

# III. DECISION LOGIC

When executing a task, agent must follow:

1. Check core/GENESIS-LAWS.md
2. Check core/SYSTEM-INPUT.md
3. Check core/OUTPUT-STANDARD.md
4. Execute task
5. Validate output

---

# IV. LOW-RISK CONTINUATION RULE

If uncertainty exists:

- If risk is LOW → continue execution
- If risk is MEDIUM → choose safest valid option
- If risk is HIGH → escalate to HUMAN

Agents must NOT freeze on low-risk ambiguity.

---

# V. TASK INTAKE SYSTEM

All tasks must originate from a structured source.

Primary source:
→ TASKMASTER.md

Task must include:
- clear objective
- defined output
- constraints (if any)

Unstructured tasks must be normalized before execution.

---

# VI. AGENT SPECIALIZATION

Each agent has a narrow role.

Rules:
- no role overlap
- no duplication of tasks
- no undefined responsibilities

Agents operate as modular units.

---

# VII. AGENT COOPERATION (HANDOFF SYSTEM)

Agents do not exchange opinions.
They exchange structured outputs.

Every handoff must follow format:

INPUT:
OUTPUT:
STATUS:
RISKS:
NEXT AGENT:
HUMAN REVIEW: YES / NO

---

# VIII. DEADLOCK RESOLUTION

If agents conflict:

1. DETECT conflict
2. ISOLATE source
3. CHECK against core/GENESIS-LAWS.md
4. CHECK core/SYSTEM-INPUT.md
5. APPLY lowest-risk valid solution

If unresolved:
→ escalate to HUMAN

---

# IX. EXECUTION STANDARD

All outputs must:

- follow core/OUTPUT-STANDARD.md
- be immediately usable
- require minimal manual correction

If output fails:
→ agent must self-correct

---

# X. PUBLISHING FLOW

Nothing is published automatically.

Required sequence:

AGENT → OUTPUT → REVIEW → APPROVAL → PUBLISH

Human approval is mandatory.

---

# XI. HUMAN OVERRIDE

Human can at any moment:

- stop execution
- reject output
- modify direction
- override agent decision

System must accept human authority immediately.

---

# XII. REPORTING SYSTEM

System must generate simple reports:

- what was created
- what was changed
- what requires approval
- identified risks (if any)

Reports must be:
- short
- structured
- readable

---

# XIII. EXTERNAL CHANGE DETECTION

If environment changes:

- platform rules
- payment systems
- content visibility
- technical compatibility

System must:

- detect impact
- log change
- flag for review

---

# XIV. 24/7 OPERATION MODEL

24/7 does NOT mean constant execution.

It means:

- system is always ready
- scheduled checks run
- event-based reactions occur

---

# XV. FAILURE HANDLING

If system error occurs:

- isolate failure
- prevent propagation
- log issue
- continue unaffected processes

---

# XVI. CONTINUOUS IMPROVEMENT LOOP

System evolves through:

DATA → ANALYSIS → ADJUSTMENT → TEST

Manual fixes must lead to:
→ system improvement (not repetition)

---

# CORE PRINCIPLE

Autonomy is allowed,
but only inside controlled structure.

System must remain:
- stable
- predictable
- real
